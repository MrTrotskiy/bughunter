- FIXED a silent data-loss bug in the trail writer: `seq` was derived by a read-then-append with no lock AND used as the graph-snapshot filename, so concurrent agent-path processes overwrote each other's snapshots. The append still throws on failure and the seed read still swallows a missing file; the counter advances only after a successful append, so a throw leaves no gap.
- NAVIGATION IS NOW MEASURED. `goToRoute` emits `timings: {gotoMs, settleMs, overlayMs, snapMs, totalMs}`, mirroring `step.mjs`'s existing act-timing field names so one renderer serves both rows. Previously `payload.timings` was `undefined` on 1200 of 1200 route events across both audited runs. Parts round down and `totalMs` rounds arithmetically — four independently-rounded parts can otherwise exceed the whole and make the trail report NEGATIVE unaccounted time. `totalMs` is measured from function entry, never summed, so the gap stays visible.
- WHY THIS RANKED FIRST, measured: attributing every inter-event gap to the event that follows it, route events are 68.6% / 74.9% of wall clock on `hygge2`/`goal1` and **90.0% on `probe10`**, while the act path was already ~78% self-explained. Named `attemptMs` deliberately, NOT `actMs`: a throw before or inside the click never completed a causal window. `step.mjs` was not touched — the capture wrapper adds no call and never runs inside a causal window.

### the crawler was re-asking questions it had already answered
- FIXED the root of the repeat clusters, and it was NOT what the brief assumed. `recordProbe` filed a blocked row under the kind the act TURNED OUT to be (`click`), while a field's battery owes `fill-valid`/`fill-overflow` — the keys never intersect, `outstanding` never empties, and `batteryOwing` re-issued the element up to `MAX_BATTERY_ROWS = 8`, **overriding an existing 3-attempt limit**. One limiter silently cancelled another; the 8 in every repeat cluster was that cap, not a coincidence.
- RETIREMENT keys on PROGRESS, not on a code list: the answer signature is (probe kind, outcome). A form ladder changes kind at every rung by construction; a repeat does not. A code list was measured and REJECTED — it would have caught 16 of raw3's 45 repeats, because repeats arise from three different mechanisms (terminal code / transient-in-principle-but-terminal-in-practice / no block recorded at all). Threshold 2 is not arbitrary: `levelOf` already calls an outcome reproduced at exactly two identical observations.
- MEASURED: repeats 30% → 19.2% (raw3), 34% → 19.5% (hunt1) — 13-18% of every act freed. All 11 real `fill-valid → fill-overflow` ladders survive in both runs, as does one control whose answer CHANGED and succeeded on the second attempt. A retired control stays in the denominator with `answeredNotExplorable`; nothing leaves the count.
- The hypothesis that two 8-act all-successful templates were the GOAL.md form ladder was FALSE, and conveniently so — it understated the loss. Their `probeKind` is null: the battery never ran. Protecting "8 acts" would have protected exactly the waste.
- `DISABLED` is now recorded as the finding `control-not-operable` (GOAL.md: the control announces itself but refuses to be operated). A policy refusal is NOT a finding — it would bury the real ones.

### `disabled` was a permanent verdict; it is a state
- `fieldFacts` merged write-once, so a control seen `disabled` once was written off forever — a wizard field, a Save button disabled until dirty. `disabled`/`readOnly` are now re-read every snapshot; every genuine declaration stays write-once. `required` stays write-once for a STRONGER reason: it is read through a three-way OR including AntD's `.ant-form-item-required` wrapper, which an early snapshot can miss, so re-reading would let a later partial snapshot unset it and silently delete the `fill-empty` obligation.
- The graph-store fix alone was INERT: `dom-snapshot` wrote `disabled: el.disabled || null`, collapsing `false` to `null`, which the merge skips — "enabled" was structurally unrepresentable.
- `readOnly` was the contested classification. Its permanence belongs to the AntD widget, not the attribute; the DOM draws no distinction. Decided on asymmetry: a stale `true` deletes coverage permanently, a fresh `false` costs one honest `NOT_FILLABLE` probe.
- The re-study counterpart is TWO rules, not one — the element-block discharge AND the no-progress tail — and it must inspect the tail IN PLACE. Filtering the disregarded rows out and rebuilding the tail would let `[answer, DISABLED, answer]` collapse into a repeat that never happened: the rule against a spurious retirement would itself cause one.
- Drain flags are deliberately NOT cleared in `mergeSnapshot`; restoration comes through `batteryOwing`. Clearing them would leave the control counted open by `frontierStats` while `terminallyAnswered` still refused to emit it — the phantom-stall mode.

### the reopen telemetry reported success for a mechanism that delivered nothing
- `relocMemo.record` filed the verdict of the REOPEN, not of the act it enabled. Measured in `hunt1` seq 491-521: eight consecutive `reopen{ok:true}` each followed by `act.failed NO_INSTANCE` — a 100% success rate for eight straight deliveries of nothing. Same class as the trail that logged an act's INTENDED target and hid a wrong-control bug for seven runs.
- Now two-phase: `record()` opens a PENDING entry and returns an idempotent `settle(actOk, code)` closure — a closure rather than a keyed call because it cannot be pointed at the wrong entry, and idempotence makes a `finally` safe. A crash between phases counts as `pending`, never `succeeded`. This is a behaviour fix, not only a reporting one: the same empty container is no longer re-bought.
- Still open, named rather than hidden: `MAX_BATTERY_ROWS` caps the disabled-flip restoration at 8 rows and `raw3`/`hunt1` already sit at exactly 8, so on those carried graphs the flip re-opens nothing; and `recovered++` still counts a reopen whose act failed — the same lie in a different variable, now terminating because the memo blocks the repeat.
