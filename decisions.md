# Decisions

Founding architectural decisions for `bughunter`. Rejected alternatives logged more than accepted ones.

### 2026-07-18 — REMOVE the read-only write-firewall + WRITE-HUNT mode entirely (operator decision)
- CONTEXT: bughunter runs as a QA tool on the operator's OWN abandoned dev stand. It is MEANT to create/edit/delete/pay. The read-only firewall (`read-only-firewall.mjs`) had become dead weight that kept silently RE-IMPOSING refusals in code paths where the flag was forgotten, and `huntWrites` was defined as a strict SUBSET of `readOnly` (`huntWrites = readOnly && …`) — a per-act write window carved into a firewall the operator no longer wants installed at all.
- CHOSE: delete the firewall and every mode derived from it. `read-only-firewall.mjs` / `read-allowlist.mjs` / `judge-endpoint.mjs` deleted; `readOnly` / `refuseMutations` / `allowBenignPost` / `revealOpener` / `huntWrites` / `huntWindow` / `huntStrict` / `createBudget` / `graph.huntCreates` stripped from `step.mjs`, `recon-run.mjs`, `whats-new.mjs`, `stateful-step.mjs`, `collect-loop.mjs`. `resolveWritePolicy` is NOT re-homed as a wrapper — `explore-policy.exploreAllArmed` already IS "explore-all armed or not", so the drivers call it directly and the redundant indirection dies with the module it named.
- CONSEQUENCE: EXPLORE-ALL is the ONLY write posture. Default (unarmed) = the deterministic `danger-floor` still declines the CLICK on destructive/auth/payment/communication and the NAVIGATION to a danger route; nothing else is gated, so an ordinary write reaches the server. Armed = those refusals lift and the ONE rail is `explore-policy`: another user's content is edited-with-restore, NEVER destroyed (`FOREIGN_DESTROY`); an account is deleted only if THIS run created it (`ACCOUNT_PROTECTED`). `hunt-gate.mjs` + `huntMarker` SURVIVE — the `HUNT-<runId>` marker is what makes "ours vs theirs" decidable, so the foreign-content rail depends on it.
- REJECTED — keeping the firewall behind a default-off flag: a flag nobody sets is untested code that still runs in every act path, and its historical failure mode was exactly the silent re-imposition the operator is removing. A gate that only ever fires by accident is not a safety feature.
- REJECTED — replacing it with a narrower guard (e.g. abort only obvious write verbs): explicitly out of scope. The operator asked for removal, not substitution; adding a smaller net would recreate the same class of forgotten-flag refusal with less coverage and no test.
- REJECTED — weakening the read-only/hunt tests into vacuous passes: `tests/CLAUDE.md` doctrine says a test that cannot fail is worse than none. They were DELETED. Cost, stated honestly: the WRITE-HUNT tests were the only LIVE end-to-end proof of the ownership rail. Live ownership coverage now rests on `portal-ownership.test.mjs` + `hunt-cleanup.test.mjs` (marked-only delete), the policy itself on the unit `explore-policy.test.mjs`. A live explore-all foreign-destroy test is the honest follow-up gap.
- KEPT — `danger-floor.mjs` intact including `mutationFloor`/`MUTATION_VERBS`, now with no lib consumer. It CLASSIFIES (labels write-ness) without gating, which is its documented role; deleting a tested classifier was out of scope.

### 2026-07-18 — NODE-LOOP HUNT (SUPERSEDED by the read-only removal above: hunt mode no longer exists; the ownership rails it introduced live on in explore-policy) (additive-only) — the scriptable mutation-collection loop (CTO/Fable-validated; RE-SCOPES the 2026-07-18 "deterministic CRUD driver" rejection at line ~62 + the write-hunt-mode memory "node loops never hunt")
- CONTEXT: the operator's standing goal #3 — "a control SCRIPT that loops until ALL UI + endpoints are collected, WITH create/edit/delete" (a QA agent cannot test a write-heavy app read-only). Today hunt is threaded ONLY through the agent path (`whats-new`); the node-loop (`recon-run`) is read-only (`actStep(..., { refuseMutations: readOnly })`, never `huntWrites`), and `certify-loop` wraps that read-only crawl. So there is no SHELL-INVOCABLE loop that mutates. This DIRECTLY RE-SCOPES two prior decisions (surfaced by the CTO, not overridden silently): the line-~62 rejection of "a deterministic CRUD driver" and the `write-hunt-mode` memory's "the node/stateful loops never hunt".
- CHOSE (Option A, single hunt-drain): thread the EXISTING, already-security-reviewed hunt machinery (`actStep`'s `{ huntWrites, marker, runCreatedAccount, huntWindow }`) into `recon-run`'s `crawl`+`persistentStep`, armed by the SAME `BUGHUNTER_HUNT`+`BUGHUNTER_RUN_ID` as `whats-new` (`resolveWritePolicy`). `recon-run --hunt` then drains the frontier+routes committing SAFE own-content mutations. The "loop" is `reconLoop` draining ONCE to DRAINED — a legitimate COLLECTION stopping condition (exact drain over the known denominator), NOT a completeness proof. The RE-SCOPE that threads the needle: the node-loop does ONLY the NON-SEMANTIC subset — ADDITIVE own-content (create/comment/like: single-control, no ownership needed, the firewall allows them) + the OWNED-SINGLE-CONTROL case (edit/delete-own gated by the DETERMINISTIC `ownsTarget` marker rail). It EXCLUDES the semantic parts the line-~62 rejection was really about — realistic fill quality, delete-with-confirm judgment, read-vs-mutation classification — which STAY on the agent path (model split preserved: the script clicks, the agent judges). Safety holds WITHOUT the LLM because the decisive rails are deterministic + driver-agnostic: `ownsTarget` reads the marker live off the DOM, the firewall gates destructive HTTP method/path on `hunt.owned`, auth + account-deletion are hard-gated in `actStep`. REQUIRED companion (the one real new risk — unbounded create spam with no judgment): a CREATE/mutation BUDGET `BUGHUNTER_HUNT_CREATE_BUDGET` (default ~15), tracked in `graph.huntCreates`, enforced in the NODE-LOOP's `persistentStep` (NOT `actStep` — keeps the shared causal primitive untouched; the agent path stays LLM-bounded) so once the budget is spent the crawl continues read-only. Chao2 certification stays a SEPARATE read-only overlay (Option C, later) — never DURING hunt, because Chao2 assumes a CLOSED population (Chao-Colwell 2017) and mutation opens it (a create exposes new UI, a delete removes re-detectable items), so a completeness number over a mutated surface is statistically invalid.
- REJECTED — Option B (exclude hunt-created instances from Q1 by tagging + subtracting): does NOT fix the real defect (Chao2's closed-population assumption is violated by DELETE-deaths + create-exposed-UI, not just the count), adds a SECOND identity (hunt-tag) into the completeness path (violates single-source-of-truth), and leaks a hidden statistical bias the operator can't see. Certification must simply not co-occur with mutation.
- SECURITY-HARDENED (security + bughunter review, all findings fixed pre-commit): (H1) the node-loop, WITHOUT the LLM judge, could commit an UNOWNED destructive write via a benign METHOD+PATH (a GraphQL/RPC `POST /graphql {deletePost}`) that slips BOTH the name gate (`requiresOwnership`, nameless) AND the firewall's destructive method/path gate — rawcaster's own `/deletenugget` is path-caught, but a site-agnostic tool must not rely on that. FIX: the node-loop passes `huntStrict` to `actStep`, which (via `hunt-gate.inOwnableItem`) declines to arm the write window for an UNOWNED control that sits inside an ownable content item — so only owned writes + a create/composer (not in any item) commit. This also declines like/comment on OTHERS' items on the node-loop (their endpoint is still collected via OWN content; the agent path, with judgment, does the nuanced case) — the deliberate conservative price of a judge-free loop. (budget=0) `Number('0'||15)===0` made an explicit `BUGHUNTER_HUNT_CREATE_BUDGET=0` invert to UNBOUNDED; fixed to `createBudget <= 0 || spent >= createBudget` (non-positive → hunt OFF). (M1) the budget now counts PER committed write (2xx/3xx non-GET), not per act, so a multi-write control can't undercount the cap. (L1) `graph.huntCreates` is coerced `Number.isFinite && >= 0` at both load and use, so a corrupt/tampered persisted value can't disable the cap. Guards: node-loop-hunt.test.mjs (D) uses `otherFirst` so the DRILL_PER_LIST representative is another user's row — the benign-path RPC delete is refused (forbiddenHits===0), revert-proven (drop huntStrict → it commits → reds); (C2) budget=0 → committed 0, revert-proven.
- REJECTED — a deterministic confirm-modal affirmative-picker in the node-loop. Measured live: rawcaster styles the delete-confirm "No" as `.ant-btn-dangerous` and "Yes" as `.ant-btn-primary`, so any "click the primary/danger button" heuristic clicks the WRONG one, and modal-recognition is exactly the forbidden heuristic-fallback. A single `actStep` clicks ONE control, so an autonomous node-loop delete-with-confirm simply does NOT fire the delete (safe no-op, honestly recorded as unreached) — delete-with-confirm STAYS agent-driven. If ever built, the affirmative selector must be an AGENT-supplied arg (like `--prefill`); the decision stays with the agent, only the same-causal-window mechanism lives in `actStep` (a confirm IS an extension of the same cause → attributing the DELETE to the Delete control is correct attribution).

### 2026-07-18 — INC.2 PORTAL-MENU IDENTITY — fold the action NAME into the template selector (CTO/Fable-validated, measured)
- CONTEXT: the reveal-backfill fix below shipped green on its fixture but did NOT reach the live rawcaster portal Delete (NO_INSTANCE, revealed=0). A read-only live probe found the real defect is UPSTREAM of reveal stamping — IDENTITY. A body-portal dropdown (AntD `.ant-dropdown-menu-item`, also Radix/MUI/HeadlessUI) mounts its items into a BARE class/id-less `<div>` appended to `<body>`, DETACHED from the trigger. `buildPath(el,false)` then emits the IDENTICAL structural templateSelector for EVERY action, and `rowKey()` returns null (a menuitem `<li>` has no li/tr/[role=row] row ancestor) so the instanceKey falls to the open-order `#N`. MEASURED live: Delete/Edit/Share/Block/Report/Become-a-Fan/… ALL collapse onto ONE template, and different actions from different triggers COLLIDE on the same `#N` (4 collisions in a single open — Share-Link#1 == Live-events#1). `mergeSnapshot`'s key-dedup then DROPS the loser, so the connectome (the template→endpoint map `addTrigger` builds with `from=element:${templateId}` — the Phase-2 product) lumps N endpoints on one node and loses reveal paths. Reveal-backfill CANNOT fix this: it stamps a path onto an identity that is itself broken.
- CHOSE: Option B — fold the menuitem NAME into the TEMPLATE selector only (`dom-snapshot.mjs templateSelectorOf` appends a synthetic ` @menu(<action>)` suffix for a detected portal menuitem). Each named action becomes its own template (distinct `addTrigger` edges → distinct endpoints in the connectome) AND — because the `#N` group counter keys on templateSelector — its own `#1` instance (the positional COLLISION is fixed as a side effect, no separate mechanism). `instanceSelector` (a live `page.$` query arg) is UNTOUCHED, so `resolveHandle`'s stale-selector → durable role-name fallback still reaches the item once its trigger re-opens the dropdown (reveal-replay). Detection is structural + snapshot-computable (`isPortalMenuItem`: role=menuitem|in-[role=menu] AND rowKey===null AND mounted under a bare `<body>`-child portal). Name normalization: a semantic-enum `value` (`/^[A-Z_][A-Z0-9_]*$/`), else aria-label, else textContent with a trailing count/badge stripped ("Become a Fan (12)" → "become a fan"), lower-cased, capped at 40 chars (above → DON'T fold, honest under-fragmentation over a per-render explosion). schemaVersion 2→3 + `loadGraph` legacy reset (a scheme-2 positional `#N` must not co-mingle with a scheme-3 name key). The ` @menu(...)` suffix is deliberately NON-CSS (can never be a live selector); verified NO consumer passes templateSelector to `page.$`/`getByRole` (identity-only: assignId keys, preVisible key, completeness sig, groupN counter). `identity-diff` is a MEASUREMENT not a green gate — proven 0 folds on hunt-social-app/param-feed-app/portal-app (byte-identical off portals).
- REJECTED: Option A — instance-level name key only (one menuitem template, name-keyed instances). Restores reachability but LEAVES the connectome collapsed: edges are template-keyed (`addTrigger` `from=element:${templateId}`), so Delete's DELETE and Share's POST still both hang off one menuitem node — the Phase-2 product stays wrong while the crawl looks green. Duplicated-truth-adjacent (the connectome lies by omission).
- REJECTED (DEFERRED) — merge-time TRIGGER-SCOPED instanceKey (bind a portal item to the acted "…" trigger via `preVisible`). CTO: do NOT build speculatively. Template-fold alone restores connectome distinctness + single-representative reachability (DRILL_PER_LIST-acceptable — one Delete representative, not Delete×50-posts). Per-row Delete-of-A vs Delete-of-B disambiguation is a separable follow-on, foldable into this same schema bump IF a later measurement demands it.
- REJECTED — dropping the `p.id`/landmark guard in `bodyPortalRoot` to also fold in-flow menubars. Keeps the re-key maximally conservative (only bare-wrapper body portals, exactly the proven live case) → guarantees zero churn on existing fixtures. An id'd-wrapper portal in some other app won't be caught yet; generalize only when a second pattern needs it (CTO inflection point).

### 2026-07-18 — REVEAL-BACKFILL preVisible transition — in-app-state reach for mount-on-reveal controls (Fable design)
- CONTEXT: the largest remaining coverage residual after the live rawcaster run — 69 of 74 unreachable controls are IN-APP-STATE (behind dropdowns/modals). Root cause (Fable-sharpened): a PORTAL dropdown menuitem (rawcaster/AntD Edit/Delete) MOUNTS on open — it is NEVER in the DOM while the menu is closed, so it can only ever be captured `visible===true` → the write-once `hiddenWhenSeen` (set = `el.visible===false` at first capture, graph-store) is STRUCTURALLY always false for it → `fillRevealIfHidden` never backfills a reveal path → the frontier hands it out with no reveal → cold re-navigation can't reconstruct "open dropdown → click it" → NO_INSTANCE. `hiddenWhenSeen` (write-once, first-ever capture) is the wrong provenance signal for a mount-on-reveal control; the panel-reach mechanism it was built for handles `display:none` controls that ARE captured hidden.
- CHOSE (Fable Option a — per-act transition, causal-locality analog): the correct "THIS opener revealed it" signal is measured in THIS act's own window, exactly like the causal token measures requests. `actStep` captures `preVisible` (the set of controls VISIBLE immediately before the act, a read-only `snapshotDom` under `__idle__` before `beginCause`, gated on a revealPath being present) and threads it into the stamped `mergeSnapshot`. `fillRevealIfHidden` supersedes the write-once flag: `revealedNow = preVisible ? !preVisible.has(key) : inst.hiddenWhenSeen` (key = templateSelector::instanceKey) — a control not visible before but visible after was revealed by this act. `preVisible` is threaded for EVERY actStep caller (whats-new/persistentStep/statefulStep all pass a revealPath), so the `hiddenWhenSeen` fallback governs only a DIRECT mergeSnapshot caller (unit tests). The panel-reach OUTCOME is preserved because preVisible AGREES with hiddenWhenSeen for a control hidden immediately before the act; they diverge only for a sibling made visible by the reveal-replay prologue, where preVisible correctly declines the backfill (that sibling gets its true [opener] path when its own opener is acted — no coverage lost; a safe re-attribution, panel-reach.test.mjs stays green). NOT a byte-identical no-op for the node loop.
- REJECTED (Fable): (b) a per-route `baselineVisible` set — coarser, can't attribute WHICH of N openers revealed a control, new persistent route-node state; (c) re-derive hiddenWhenSeen from CURRENT visibility — destroys the signal (the control IS visible at the fill point, so it never fires; thrashes when sampled elsewhere) + re-opens the genuine-coverage false-positive. REJECTED (now): a MutationObserver DOM-causality signal for the auto-appear residual (deferred graph.states menu-model territory).
- RISK VERDICT (Fable, explicit): LOW-RISK TARGETED FIX — does NOT require the deferred `graph.states{}` State-node. The reveal path is already a SEPARATE addressing dimension (never identity); the gap was PROVENANCE, not a missing depth/identity mechanism. Closes the depth-1 portal/mount-on-reveal residual that dominates the 69; cross-state dedup + behind-state COUNT remain correctly deferred to graph.states (reporting refinements, not reach). Composes with the shipped `ownsViaReveal` portal-ownership fix (applyReveal re-opens the portal, ownsViaReveal authorizes it).
- INVARIANTS (all pass): no phantom edge (the pre-snap is a read-only evaluate under `__idle__`, before beginCause — no causal window, no addTrigger; the backfill runs after endCause); no second identity (`preVisible` is a transient per-act Set, never persisted / never an id input); denominator (a truthful unreachable→reachable increase, no new template/instance); no bogus Phase-2 path (guarded by el.visible + the genuine-coverage guard + the stamp gate + replay-validation: a wrong path fails REVEAL_STALE → honest unreachable, never a reachable lie). Named residual: a control that AUTO-appears (deferred timer/animation) coincidentally in the opener window false-positives — bounded by the genuine-coverage guard + caught by replay validation.
- VERIFY: `tests/unit/reveal-backfill.test.mjs` on a KNOWN instance (the rawcaster case) — preVisible transition backfills a first-captured-visible menuitem (revert to hiddenWhenSeen → reds, verified); NO backfill when already visible pre-act (the coincidental-control guard); legacy hiddenWhenSeen fallback still works. Plus `tests/live/portal-menu-reach.test.mjs` (a mount-on-reveal portal menuitem is revealed + REACHED via reveal-replay — GET /copy caused). ONE thing to measure LIVE before trusting Phase-2 output: on an agent-path `/recon` run, count backfilled reveal paths that FAIL replay (REVEAL_STALE) vs succeed — many failures ⇒ the "…" is hover-triggered (a separate primitive) or deferred-render pollution.

### 2026-07-18 — WRITE-HUNT portal-dropdown ownership (live rawcaster finding: detached-portal delete)
- CONTEXT: the first LIVE hunt CRUD run on rawcaster (Stierlitz) proved CREATE (a real HUNT-marked nugget committed via `POST /rawcaster/addnuggets` + prefill + reveal), but DELETE could not self-clean: rawcaster (AntD) renders a row's Edit/Delete dropdown as a PORTAL appended to `<body>`, structurally DETACHED from the post card. `hunt-gate.ownsTarget` walks `closest(ITEM_BOUNDARY)` from the delete button → finds no marker in its DOM ancestors → fails closed → a LEGITIMATE delete of OWN content is aborted (`hunt-unowned-blocked`). Not a false positive — a real architectural gap for portal-UI dropdowns.
- CHOSE: `hunt-gate.ownsViaReveal(page, graph, revealPath, marker)` — a portal control's ownership is the ownership of the control that OPENED it. The reveal path's LAST hop is that trigger (the row's "…" more button, which IS inside the card); resolve it from its stored `instanceSelector` and run the SAME marker check on it. `step.mjs` uses it as a FALLBACK only when `ownsTarget(handle)` fails and a revealPath exists. FAIL-CLOSED on every path (no reveal path / unresolvable trigger / trigger not in a marked row → false). Safe because the dropdown opens FOR the row whose trigger was clicked, so the trigger's post IS the delete's target — authorizing by the trigger's ownership never authorizes a different row.
- REJECTED: widening ITEM_BOUNDARY to catch the portal (re-opens the H2 sibling-marker leak); inferring the target row from the portal's screen position (fragile, not structural); an in-memory created-id store (the graph + marker already are the record). REJECTED for now: the reveal-BACKFILL gap that also blocked the live delete (a dropdown menuitem that is a KNOWN template gets no `reveal` path from mergeSnapshot when it was first captured VISIBLE) — a deeper mergeSnapshot reveal-provenance fix, scoped separately; ownsViaReveal is correct + committable independently (it activates once a reveal path exists).
- VERIFY (revert-proven, `tests/live/portal-ownership.test.mjs` on a new `portal-app` fixture — a `#portal-menu` Delete detached from the post cards): ownsViaReveal via OUR marked row's "More" trigger → owned; via ANOTHER user's row → refused; empty reveal path / no marker → fail-closed. All hunt-writes + suite green.
- OUTSTANDING (live side-effect): the test nugget "Testing rawcaster QA hunt [HUNT-huntlive1]" is still live on Stierlitz (the reveal-backfill gap blocked the automated self-clean this run) — needs a manual delete until the backfill fix + a hunt-cleanup CLI land.

### 2026-07-18 — GOAL 5: Chao2 VARIANCE SOURCE + param-key fix (the completeness-certification enabler)
- CONTEXT: the Chao2 mark-recapture oracle (`completeness.mjs`) was built but INERT — a dedup crawler drains the SAME frontier deterministically every run, so two crawls give identical detection sets → Q1=0 → a degenerate "100% complete". To CERTIFY "we're at X% of the estimated reachable surface" it needs run-to-run VARIANCE (≥2 crawls that genuinely differ) + a cross-run-stable item key.
- CHOSE (variance): a deterministic SEEDED SHUFFLE (`frontier.seededOrder`, mulberry32 over an FNV hash of `BUGHUNTER_SEED`) re-permutes the frontier emission order in nextBatch/navBatch. With a per-run `--steps` budget, two DIFFERENT-seed crawls explore different subsets → Q1>0 → Chao2 becomes informative. Default (no seed) returns the ascending order UNCHANGED (byte-identical, resumable). Threaded through frontier-cli (`BUGHUNTER_SEED`, agent path) + recon-loop/recon-run (node path). Reorders emission ONLY — the drain predicate + honest denominator are untouched.
- CHOSE (cross-run key): `completeness.routeItemKeys` keys a concrete `:param` instance by its `paramInstanceOf` PATTERN, not the concrete route — so run A landing `/nugget/1` and run B `/nugget/2` count as ONE item (the pattern), not two unique detections. Without this the shuffle would make every distinct concrete a Q1 singleton and Chao2 would grossly OVER-estimate the dark tail. (The GOAL-2 inflection point, now resolved.)
- REJECTED: `Math.random`-shuffled order (non-reproducible — a re-crawl must be resumable + a seed must reproduce); shuffling what is WALKABLE (would change the denominator — only the ORDER may vary, never the set); a wall-clock/entropy seed (breaks determinism).
- BUILT (the AUTONOMOUS self-terminating loop the operator's `/goal` demands — "loop until EVERYTHING is collected"): `certify-loop.mjs`. `certifyDecision(graphs, {target, minRuns})` certifies when the Chao2 estimate CONVERGES (Q1=0 — the shuffled re-crawls found nothing unique) OR both dimensions clear the target; `certifyLoop({runCrawl, target, maxRuns})` re-crawls with a rotating seed until certified, else stops HONESTLY at a maxRuns runaway cap (never a faked 100%). The stopping condition is a MEASUREMENT, not a fixed --steps guess. Injectable runCrawl → unit-tested without a browser (converges-at-run-2 + always-discovering-caps, revert-proven). CLI wires runCrawl to a seeded in-process recon-run per sample. Still deferred (pure polish): `report --discovery-curve`.
- VALIDATED LIVE on rawcaster (authed read-only node-loop): the manifest seeded 60 declared + 23 `:param` patterns; the GOAL-1 negative-control probe DETECTED 16 client-404 phantoms (/contact, /feed, /groups, /goals, …) + 16 redirects → the honest denominator collapsed the inflated 60 to 28 COLLECTABLE (15/28 = 54% collected in a bounded node run; the 13 visited-empty + 0/23 param-collected are the known node-loop read-only sparsity — POST-content aborted without the agent read-allowlist; the agent path + hunt collect the rest). Before GOAL 1 the same site read 23/64=36% over an inflated denominator; now the denominator is honest.
- VERIFY (`tests/unit/chao2-variance.test.mjs`): seededOrder — no-seed identity, same-seed reproducible, a seed permutes (set preserved, input not mutated), different seeds differ (revert: ignore the seed → the permute assertion reds); routeItemKeys — a `/nugget/1` proxy keys by `/nugget/:id`, the concrete key is NOT emitted (revert: drop the paramInstanceOf key → reds). Frontier + completeness suites stay green.

### 2026-07-18 — GOAL 2: param-instance harvest — collect the dark `:param` patterns (biggest section delta)
- CONTEXT: the route-manifest seeds `/nugget/:id`, `/user/:handle` etc. into the denominator as `param-pattern` (counted, NEVER navigated — a literal `:id` path 404s — 0 collected). On rawcaster the `:param` patterns are ~41% of the section denominator at 0%. GOAL 2 harvests ONE concrete instance per pattern, genuinely visits it, and flips the pattern from 0-collected to collected-via-representative — the biggest coverage delta.
- CHOSE (reconcile): a STRUCTURAL `matchParamPattern(graph, concreteRoute)` (graph-store) — align a concrete to a declared `:param` node by SEGMENT STRUCTURE (same segment count; a pattern `:name` slot matches any concrete segment; literals must be equal). `tagParamInstance` writes an additive `paramInstanceOf` link (REPORTING metadata, NEVER identity — route identity stays routeKey). Wired at the three edge-free reach points: `harvestRoutes` (fallback a[href]), `visitRoute` (a seeded concrete), `actStep` (a drill-act that navigated to a concrete, post-endCause). Agent path gets it free (whats-new/route-cli call the same harvest/visit).
- CHOSE (census for param): a NEW param-branch in `harvestRoutes` folds a concrete matching a declared pattern into the FIRST concrete representative's siblings — because `toUrlPattern` masks only digit/uuid/hex, a STRING-keyed `/user/alice` never shares a `toUrlPattern` with `/user/bob`, so the existing `representativeFor` census would visit BOTH; the structural fold bounds string-keyed lists to one visit too.
- CHOSE (report): `routeCoverageOf` gains `paramCollected` (a pattern whose `paramInstanceOf` representative is visited-with-content) + `collectedTotal`/`collectableTotal` (fold the param patterns into the headline). The concrete `paramInstanceOf` proxies are EXCLUDED from the static `sections` count (no denominator inflation); the pattern node NEVER leaves the denominator (flips buckets, never removed).
- REJECTED (Fable): navigate the `/x/:id` LITERAL (404/garbage — a `:param` string is not re-navigable, the reason routeKey keeps concrete paths); a `toUrlPattern`-equality reconcile (silently drops string-keyed params — the live risk Fable flagged: `/viewprofile/:username` → `/viewprofile/bob` is unmasked); counting each concrete as its own section (inflates the denominator by every drill-landing); `paramMask` (a colon-name masker) — the structural matcher is strictly more robust (handles string keys) with no new masker.
- DEFERRED (Fable, separable sub-increment): virtualized-list `aria-rowcount` honest counting (LOCK 3). GOAL 2's coverage target is met by collecting ONE representative per pattern; the exact mounted-vs-declared sibling tally is honesty polish, built only when a live list proves the undercount material AND lacks aria-rowcount.
- HARDENED (bughunter review MUST FIX): `tagParamInstance` guarded the pattern/already-tagged cases but NOT `node.declared === true` — a manifest-DECLARED static sharing a param's shape (`/user/settings` beside `/user/:handle`; every 1-segment static under a `/:slug` catch-all) matched structurally, got tagged `paramInstanceOf`, and VANISHED from `sections` → denominator collapse + fabricated pattern coverage (invariant #3, the #1 forbidden). FIX: `tagParamInstance` refuses a `declared === true` node (a declared static is always its own section). Plus a most-literals-first tie-break in `matchParamPattern` so a concrete is credited to the most SPECIFIC pattern, not a bare catch-all. Both revert-proven (drop the declared guard → the `/user/settings` collision test reds).
- VERIFY (revert-proven, `tests/live/param-instance.test.mjs` on `param-feed-app` with a bundle declaring `path:"/nugget/:id"` + `path:"/user/:handle"` + a DECLARED `path:"/user/settings"` static collision): NUMERIC `/nugget/:id` collected via ONE visited `/nugget/1` (census, others folded); STRING-keyed `/user/:handle` linked via the structural matcher to `/user/alice` (census folds /user/bob) — drop the structural fold → /user/bob visited separately → census reds; paramCollected===2, collectedTotal folds them in, proxies not counted as sections. ### 2026-07-19 — INC.7: container re-entry, measured before wiring
- CONTEXT: three of six target flows have no URL (/groups is a client-404, /events and /chats redirect to
  /dashboard) — their functionality exists only as dashboard modal state. The stateful driver's only recovery
  when a control does not resolve is to re-navigate its route, which CLOSES the modal; 141 of 180 navigations
  in inc6d produced zero acts. It retries the ROUTE dimension when the missing dimension is STATE.
- CHOSE: build `reopen-policy` + `reopen-container` + `flow-probe` and MEASURE, wiring nothing into the
  driver. The measurement decides whether the 476 recorded reveal paths are re-entry routes or merely
  provenance. `reveal-replay.mjs`'s REVEAL_PROVENANCE_ONLY guard is untouched — this is a separate, stricter
  admission path whose soundness comes from VERIFICATION: after the walk the intended target must re-resolve,
  so a wrong path degrades to "still unreachable" and can never produce a mis-attributed act.
- CHOSE: replay the SUFFIX of a path, shortest first, not the whole path. The last hop is the immediate
  opener by construction. Measured: 37 distinct last hops cover all 87 unreachable-with-path instances, while
  a recorded-depth cap covers far less and would exclude the headline case outright (the live `Create` sits at
  depth 4 behind `See Translation` and `Read Out`, transparently not on the path to a Create modal).
- THE GATE TOOK THREE VERSIONS and the history is the lesson:
  v1 refused any hop whose recorded endpoints classified as a write. On this target that is unanswerable
  lexically — `POST /rawcaster/addnuggetview` is a VIEW COUNTER that `endpoint-class` reads as a write via the
  verb `add`, so a menu opener, a nav tab and a counter all looked mutating; `influencerlist` is a plain list
  that falls to the non-GET fallback, which on a POST-for-reads app is the normal shape of a read. Result: 12
  of 37 openers refused, catching ZERO irreversible acts, including `add` → `Create` refused on its NAME with
  zero recorded requests.
  v2 (CTO) dropped the write axis entirely for irreversibility + a budget. Rejected on review: `dangerFloor`
  is `safe` for all 34 openers, which proves it has no resolving power on that population rather than that the
  population is safe — "report Report Abuse" and "block Block User" are in neither the destructive nor the
  communication set yet are outward-facing and effectively irreversible.
  v3 (shipped): refuse on the decidable axis (danger floor, account deletion, destroys/edits existing
  content), and keep ONE name-axis refusal that is EVIDENCE-CONDITIONED — a hop whose own instance was acted
  cleanly is admitted whatever its name suggests; a hop with no clean act behind it and a non-safe name is
  refused. Per INSTANCE, never per template: `markInstanceUnreachable` propagates to the node only for
  instances[0], so tpl 515 and 521 both read clean at node level while carrying a failed instance.
- REJECTED: patching `endpoint-class` so `addnuggetview` reads as a read. It genuinely inserts a view record,
  and no lexical rule separates it from `createlist` — bending a shared classifier to serve one consumer's
  different question would corrupt the write-surface headline in the direction already burned once.
- CHOSE: the ownership rail runs at the HANDLE SEAM in the walker, not in the policy. Ownership is proven live
  off the resolved handle and `admitHop` is pure, so it structurally cannot compute it. A hop whose verdict
  needs a restore bracket or a re-login is refused rather than performed unbracketed — plumbing has neither.
- BUG FOUND IN REVIEW: I wrapped `inOwnableItem` in `.catch(() => false)`. That function fails CLOSED by
  returning TRUE (treat as in-item → block the unowned write); the catch inverted it, leaving ownership NONE
  and full rights. The rule was not duplicated, but its FAIL DIRECTION had diverged — the same defect class.
- MEASURED: 19 of 87 instances recoverable with the full gate including the ownership rail (26 without it —
  the 7 are seven places the plumbing was about to click another user's content, a finding rather than a
  cost). The acceptance criterion was then corrected: the 87 denominator answers "would wiring recover much",
  NOT "are the breadcrumbs paths", because it includes instances our own policy declined to attempt. The
  architectural verdict is over ATTEMPTED walks only, reported separately.

### 2026-07-19 — the FOREIGN_ADDITIVE gap: outward acts against real people
- FOUND while re-examining the operator's own rule set at his request. `decide()` on another user's content
  returns allow=true for Report Abuse, Block User, Send Message, Add Friend, Invite AND Video Call — all as
  FOREIGN_ADDITIVE, justified as "creates nothing of theirs to lose". That is true of a like or a comment and
  false of a report: the category conflates ADDITIVE-TO-THE-DATA-MODEL with HARMLESS-TO-A-PERSON, and on a
  live social app with real users the second is what matters.
- WORSE: under explore-all the `REFUSED.has(floor)` check in `step.mjs` sits in the `else` branch, so
  `communication` is never consulted in the mode we actually run — despite being placed in REFUSED precisely
  because "initiating a real-time call is an IRREVERSIBLE OUTWARD side-effect".
- CHECKED whether it had already happened: across inc6b/inc6c/inc6d the crawler clicked "Block User" and
  "Report Abuse" dozens of times. NO block/report/abuse endpoint exists anywhere in the graph; the only
  related request ever fired is `POST /rawcaster/listallblockedusers`, a read. Nothing reached a real person —
  and the reason is that the confirmation dialogs were never completed. We were protected by the same
  modal-completion failure this increment exists to remove.
- OPERATOR DIRECTION (his own revision): the policy is ENVIRONMENT-TIERED. dev/staging — full freedom,
  including another user's content, because there the "other users" are fixtures. prod — ALWAYS restricted;
  the default is the ABSENCE of a grant, and the operator opens specific capabilities explicitly.
- OPEN, and the line I intend to hold: outward-facing acts (email, SMS, payment, a real-time call, a report
  against a person) stay refused on EVERY tier, because they leave the system identically whichever
  environment fired them. Under consensus review; not yet implemented.

### 2026-07-19 — capture frames keyed by probe, not by template
- CONTEXT: frames and body dumps were named `t<templateId>-before.png`. That is unique only while a
  template is acted at most ONCE per run — a property the frontier's drain predicate happened to
  guarantee (one act per control, ever), never a property anyone chose. The project is moving toward
  studying an element with SEVERAL probes rather than touching it once; at the second probe the name
  collides and the write silently overwrites the first frame. The evidence for the earlier probe is
  gone with NO error — the worst failure shape for a tool whose whole output is evidence.
- CHOSE: a monotonic per-process capture ordinal AHEAD of the template id — `a0001-t7-before.png`.
  Ordinal first so a directory listing sorts in capture order; the template id kept so a frame stays
  greppable by control. The same stem covers the redacted request/response body files, which carried
  the identical collision.
- CHOSE: assert the SHAPE (`^shots/a\d{4}-t<id>-before\.png$`) in the live test, not a fixed ordinal.
  The counter is per PROCESS, so any test that acts ahead of this one would otherwise break on a
  number that carries no meaning.
- REJECTED: a per-template counter (`t7-2-before`) — reads as "template 7, instance 2" in a codebase
  where instances are a real concept, and it loses the global capture order the trail is read in.
- REJECTED: a timestamp suffix — two frames inside one act can land in the same millisecond, so it
  reintroduces the collision it was meant to fix, and sorts worse than a zero-padded counter.
- UNCHANGED, deliberately: screenshots stay gated behind `BUGHUNTER_VIEW=1` and `/recon` does not
  export it. Logs always, frames only in view mode — the operator's standing instruction. This changes
  what a frame is NAMED, never when one is TAKEN.

### 2026-07-19 — outcome observables: REFUSED and INERT told apart on evidence
- CONTEXT: an act produced exactly two observations — the requests it caused and the elements it revealed.
  Everything else was inferred, and the inference that mattered most, "the form was refused by validation",
  had no evidence behind it. It was asserted across six runs and then measured to be flatly wrong: the
  submit had never been clicked (INC.6b), so nothing had ever been validated. A verdict nothing can
  contradict is not a verdict.
- WHY IT IS COVERAGE, not tidiness: a submit that fired nothing because the page rejected it is a WORKING
  form we failed to satisfy; one that fired nothing because the control is dead is weight in the
  denominator. Both scored identically before this.
- CHOSE: three refusal tiers read most-standard-first — WHATWG constraint validation, then ARIA
  (`aria-invalid` + `aria-errormessage`), then framework markup (`.ant-form-item-explain-error`). A silent
  tier costs nothing, and a target that answers on tier 1 needs no framework knowledge at all.
- CHOSE: a SUCCESS channel (live regions) that stands on its own rather than as a fallback beneath the
  refusal tiers. MEASURED on the live target: the Create Event form answers on NO refusal tier at all and
  accepts a completely EMPTY submit with `POST /api/meetings-events -> 201`, returning a real id; its only
  signal of any kind is a live region reading "Event was successfully created". A target silent on tiers
  1-3 is normal, not broken. (The empty-accept is also a genuine application bug — the form creates a
  record from nothing.)
- CHOSE: `:user-invalid`, NOT `checkValidity()` alone. An empty required field is invalid the instant the
  page loads, so `checkValidity()` reported a refusal on a page that had said nothing — the exact false
  inference the module exists to remove. The test caught that. Where the pseudo-class is unsupported the
  flags still ride, so an older engine loses precision about WHEN, not the reading itself.
- CHOSE: dismiss dialogs on capture rather than merely record them — an unhandled `confirm()` blocks every
  subsequent command and would freeze the crawl.
- REJECTED: inferring refusal from "no requests fired". That IS the six-run inference, and it cannot tell a
  rejected form from a dead control; `wasRefused` requires the page to have SAID something, and restoring
  the fallback is the module's own fail-on-revert lever.
- NOT WIRED: nothing consumes this yet. It is the observation surface the probe battery will read.
