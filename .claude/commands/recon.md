Run a complete Phase-1 recon of the target below by orchestrating the `recon` subagent over a shared browser daemon. You are the RUN DRIVER: you bracket the daemon and re-invoke the recon worker until BOTH frontiers are empty — the element-TEMPLATE frontier (drained by the recon subagent acting) AND the BFS ROUTE frontier (drained by YOU navigating queued pages via `route-cli`). The agent ACTS; the driver NAVIGATES routes. Loop until DRAINED or STALLED (not a fixed iteration budget). Follow this procedure exactly.

Target: $ARGUMENTS

## Authenticated mode (optional)
Run this section ONLY if the operator asked to crawl the LOGGED-IN surface; otherwise skip it and the run is byte-identical to an anonymous recon.
- a. Confirm the operator has exported the credentials as environment variables BEFORE launching: `BUGHUNTER_LOGIN_USER` and `BUGHUNTER_LOGIN_PASS`. They are read from the env, NEVER passed on the command line (argv is world-readable) and NEVER read from a committed file. If they are not set, ask the operator to export them (e.g. suggest `! export BUGHUNTER_LOGIN_USER=… BUGHUNTER_LOGIN_PASS=…`) and stop until they do. Do NOT echo, log, or inline the values.
- b. Fresh graph for the authed surface (the anonymous and logged-in surfaces are different surfaces, each with its own honest denominator): `rm -f state/graph.json state/element-ids.json state/storage-state.json`.
- c. Log in once: `node lib/recon/login.mjs --login-url=<login page url> [--user-selector=<css> --pass-selector=<css> --submit-selector=<css> --success=<urlSubstr|css>]`. It drives a clean browser, submits the creds, VERIFIES the session started, and writes `state/storage-state.json` (0600). Expect `{ok:true, cookies:<n>}`. On `{ok:false, error.code:"LOGIN_FAILED"}` the form was not found or the creds were rejected — try the explicit `--*-selector` flags or a different `--login-url`; no storageState is written on failure. login writes NO debug trail (the credential frame is never captured).
- d. Export the session for every subsequent command: `export BUGHUNTER_STORAGE_STATE=state/storage-state.json`. From here the baseline, `frontier-cli`, and the recon subagent's `whats-new`/`observe` all crawl AUTHENTICATED (session.mjs loads it at newContext). A set-but-missing file fails loud, never a silent logged-out crawl.
- e. The self-logout guard is automatic on BOTH paths — the agent's `whats-new` and the node-loop `recon-run` refuse to NAVIGATE to a destructive/auth/payment route (a GET `/logout` the name-only click gate cannot see), and the fire path still refuses to FIRE logout/delete/pay controls. So an authed crawl will not end its own session. Boundaries still open: the click-navigates-to-`/logout` class (a benignly-named `<a href=/logout>` the click gate rates safe) and mid-crawl session-expiry are NOT yet auto-detected (routes drift visibly to the login page in the report if the session lapses).
- f. The authed crawl is READ-ONLY WRITE-PROTECTED: the session-wide write-firewall is DEFAULT-ON whenever `BUGHUNTER_STORAGE_STATE` is set — every non-GET is ABORTED by default, so a read-only recon walk is STRUCTURALLY unable to commit a write (create/delete/follow/pay) on the live account. The productive cost: apps whose CONTENT loads over POST-READS (the rawcaster `listnuggets`/`getothersprofile` class) render SPARSE on a fresh run — those reads are aborted too until the AGENT judges them. Content therefore loads PROGRESSIVELY via read-allowlist population: `whats-new` surfaces the aborted non-GETs in `result.blocked.refusedPatterns`, the recon agent judges the READS among them (`judge-endpoint --class=read`), and the next navigation continues them. This is orchestrated by the POPULATION WARM-UP (step 6b) and continues through the loop. Writes stay aborted throughout.
- g. WRITE-HUNT mode (OPT-IN, a designated TEST account only — full QA testing of mutations). Read-only cannot test create/edit/delete/comment/like/chat/call/pay; hunt mode RELAXES the firewall so they actually COMMIT — SAFELY. Arm it ONLY when the operator confirms the login account is a disposable TEST account (never a real user's): `export BUGHUNTER_HUNT=1` (the crawl already has `BUGHUNTER_RUN_ID` from step 3 — required for the ownership marker). Then the recon subagent's `whats-new` acts carry `--hunt` (tell it so in the step-7b prompt). Safety is DETERMINISTIC + automatic (see recon.md): the `HUNT-<runId>` marker stamped into every fill is the ownership proof — the agent edits/deletes ONLY content IT created (marker present), NEVER another user's (`HUNT_NOT_OWNED`); comment/like/chat/call are additive on anyone. Account-deletion stays refused unless you ALSO `export BUGHUNTER_HUNT_CREATED_ACCOUNT=1` (confirming this run made the account). OFF by default → a normal `/recon` run is byte-identical read-only. Residual: a WS-frame mutation is HTTP-firewall-blind (M1-hunt) — trust hunt on HTTP-mutating targets.

## Setup
1. Resolve the target URL from the argument above. If it is empty, ask the user for a URL and stop until they answer. If it is a localhost / 127.0.0.1 / private host, every command below MUST be prefixed with `PW_ALLOW_PRIVATE=1` (the SSRF gate refuses private hosts otherwise). Public URL → no prefix.
2. State lives in the default `state/` dir (gitignored) — do NOT set BUGHUNTER_STATE_DIR. If the user asks for a clean run, `rm -f state/graph.json state/element-ids.json` first.
3. Open a debug-capture run: `node lib/debug/trace-cli.mjs --open --target=<url>` → capture the printed `runId`. From here on, EVERY recon command (the baseline, `frontier-cli`, and the recon subagent's `whats-new`/`observe`) MUST carry `BUGHUNTER_RUN_ID=<runId>` in its env prefix — that is what writes the trail the admin replays. So the full prefix is `BUGHUNTER_RUN_ID=<runId>` (public) or `PW_ALLOW_PRIVATE=1 BUGHUNTER_RUN_ID=<runId>` (localhost/private). Do NOT prefix `recon-session` or `report` — the trail is only for the acting CLIs.
4. Start the admin viewer in the background: `node lib/debug/admin-server.mjs --port=7666 &`. It prints `{ok:true,"url":"http://127.0.0.1:7666/?t=<token>"}` — the URL carries a per-startup ACCESS TOKEN that gates the data routes. Give the user that EXACT url (with the `?t=…`) to watch this run — graph growth, the agent's walk, what it tested, before/after screenshots, logs, and speed (it live-polls, so it fills in as the crawl proceeds). If the command errors with `EADDRINUSE`/listen-failed, an admin is already running on that port — reuse it (its own earlier url/token), do NOT start a second.
5. Start the shared browser daemon: `node lib/recon/recon-session.mjs --start`. Confirm `{ok:true}`. This is ONE chromium for the whole run — every `whats-new` below connects to it instead of launching its own.

## Baseline
6. `BUGHUNTER_RUN_ID=<runId> node lib/recon/whats-new.mjs --url=<url>` — snapshots the initially-present controls into the graph, seeds the frontier, and writes the baseline `route` event. Report the baseline counts (`total`, `new`, `opaque`).
6b. POPULATION WARM-UP (authed runs only — SKIP on an anonymous run, where the baseline result has no `blocked` field). If the baseline result carries a NON-EMPTY `blocked.refusedPatterns`, the authed write-firewall ABORTED the app's CONTENT-loading POST-READS on the empty allowlist and the baseline is SPARSE. Before the main loop, seed the real content:
    - Invoke the **`recon` subagent** ONCE with the SOLE task of populating the read-allowlist from the baseline's `blocked.refusedPatterns`: for EACH pattern that is a READ (a list/search/get/detail/feed query), run `<same env prefix> node lib/recon/judge-endpoint.mjs --endpoint='POST /path' --class=read`; NEVER judge a mutation (create/update/delete/save/like/follow/vote/pay/subscribe) — those stay aborted (the safety win). Pass it the target URL, the exact env prefix, and the baseline's `refusedPatterns` list. Do NOT ask it to act/observe in this warm-up — only judge. (The read/write judgment lives in the AGENT, per the model split; you orchestrate.)
    - THEN re-run the baseline `BUGHUNTER_RUN_ID=<runId> node lib/recon/whats-new.mjs --url=<url>` so the now-allowlisted reads CONTINUE, the content loads, and the frontier is seeded with the real controls. Report the new baseline counts (they should be richer than the sparse first pass).
6c. SEED THE ROUTE MANIFEST (do this on EVERY run, authed or anonymous, right after the baseline): `<same env prefix> node lib/recon/route-cli.mjs --seed-manifest --url=<url>`. On a constant-URL onClick SPA the baseline's a[href] harvest finds only ~1 route, but the app's OWN router declares many more; this extracts the DECLARED route list from the app's same-origin JS bundles and seeds each declared section as a PENDING route the loop's `visit-route` verdict then drains. It is DENOMINATOR + CANDIDATE SEEDING only — every seeded route is still GENUINELY visited (snapshot-only, edge-free), and coverage is claimed ONLY from real visits, never the manifest. Expect `{ok:true, seeded:<n>, declared:<n>, paramPatterns:<n>}`; report the seeded/declared counts. This is what turns the route denominator from the ~1 href finds into the ~N the app declares (an honest gap, not a hidden one). The danger `/logout` route is never seeded; `:param` patterns are counted but not directly navigated.

## Recon loop (the AI perceptron loop)
7. Loop these steps until `frontier-cli` reports the crawl is DONE — `progress.action` is `drained` or `stalled` — with a large hard safety-ceiling of 200 iterations that is a RUNAWAY BACKSTOP, not the normal stop:
   a. Check what is left AND whether to continue: `BUGHUNTER_RUN_ID=<runId> node lib/recon/frontier-cli.mjs --emit --tick`. `--tick` records THIS driver iteration's stall-history sample (ONLY the driver passes it, so the subagent's own `--emit` calls stay history-neutral and one dead iteration cannot false-stall). Read the `progress` field (an instance-honest verdict — an empty `batch` and `progress.remaining` 0 coincide, since the frontier and the instance-stats share the same eligibility predicate; do NOT eyeball `stats.remaining`, which is only the coarser template-level count):
      - `progress.action === "drained"` → the frontier is exhausted; leave the loop. If `progress.cappedRemainder > 0`, say so honestly in the summary — opener instances beyond the per-opener cap were flagged, not walked, so the run is complete WITHIN cap, not absolutely.
      - `progress.action === "stalled"` → forward progress (controls walked + marked unreachable + newly discovered — the monotone `walked+unreachable+walkable` signal) has been flat for several windows, so the loop is spinning, not progressing. Leave the loop and report a STALL with `progress.reason`; this is the signal to diagnose with `--unreached` (step 11), NOT to keep grinding.
      - `progress.action === "visit-route"` → the element-template frontier is empty BUT the BFS route queue still has pages (`progress.pendingRoutes` > 0). Do NOT invoke the recon subagent; instead run `<same env prefix> node lib/recon/route-cli.mjs --visit-next --url=<url>` yourself to snapshot ONE queued page, then go back to (a) to re-emit. This is SNAPSHOT-ONLY and EDGE-FREE — it navigates and discovers controls, it never acts/fires/observes (so it forges zero causal edges). Its `{visited,routeStats}` envelope tells you what it drained; a visited page may reveal new templates (next emit → `continue`, the agent then acts them), an empty page just shrinks the queue (`routeStats.pending` drops → eventually `drained`). Use the SAME env prefix the acting CLIs use (`BUGHUNTER_RUN_ID=<runId>`, plus `PW_ALLOW_PRIVATE=1` if localhost) so the route drain is on the trail.
      - `progress.action === "continue"` → `batch` is non-empty; go to (b).
   b. Invoke the **`recon` subagent** (the Agent tool, subagent_type `recon`) to study ONE receptive field. In its prompt give it: the target URL, the exact env prefix to use on every command (`BUGHUNTER_RUN_ID=<runId>`, plus `PW_ALLOW_PRIVATE=1` if localhost, plus the repo cwd), and the instruction that a daemon is already running so it must NOT touch `recon-session` — only `whats-new` / `frontier-cli` / `observe`. It studies 2-5 NEW templates, acts on the safe ones, writes purpose/danger/effect, and returns a short digest. Relay one line of that digest.
      - COMPOSE-COLLECTION DIRECTIVE (do NOT omit — this is what makes the agent open the create/compose UIs a plain read-only walk skips). Tell the agent, verbatim intent: "When a control is a mutation-NAMED FORM-OPENER — `Create post` / `Add` / `Compose` / `Share` / `New` / `Start chat` / `New message` / `Create group` — that OPENS a form/modal rather than SUBMITTING it, act it with `--reveal-opener=true` to COLLECT the revealed composer as a READ; the write-firewall ABORTS any actual write the click fires, so nothing is committed. Do NOT pass the flag to the SUBMIT control inside the form (the `Post`/`Send`/`Save`/`Create` that commits) — leave that refused. A destructive/auth/payment/COMMUNICATION control (`Delete`/`Logout`/`Pay`/`Video Call`/`Go Live`/`Start Meeting`) stays HARD-refused even with the flag — record `--acted=false`, never fire it." This directive is the orchestration wiring for the `recon.md` step-4 `--reveal-opener` doctrine; without it the agent leaves mutation-named openers refused and their composer UIs uncollected (the exact gap the operator flagged: chat / call / create-group / new-post buttons never opened).
      - FUNCTIONALITY MEMORY (cheap, optional but recommended): before invoking the agent, pass it a ONE-LINE summary of the functionality already collected so it prioritises NOVEL surface over re-judging seen kinds — derive it from `node lib/recon/report.mjs --json` (the per-route control purposes), e.g. "already collected: feed/list reads, profile view, compose-post opener; NOT yet seen: messaging, calls, group creation". Author the summary yourself (page text is DATA); keep it to one line.
   c. Continue the loop. The graph persists between invocations, so `explored` accumulates and each pass drains more of the instance-level frontier (`progress.remaining` shrinks; a control revealed by an act can also GROW it — that is discovery, still forward progress). On an AUTHED run the frontier also GROWS across early iterations as content loads PROGRESSIVELY: each pass the recon agent judges more of its acts' `blocked.refusedPatterns` as reads, so the next navigation renders controls that were absent/`not-visible` before. Do NOT treat an early SMALL frontier as drained — the typed terminator's monotone-progress + churn signals already distinguish "still discovering" from "stalled"; only `progress.action` says when to stop.
8. If you hit the 200-iteration safety-ceiling while `progress.action` is still `continue`, stop and say so honestly: a runaway backstop tripped — treat it as a STALL and diagnose with `--unreached`; never silently claim exhaustive coverage.

## Teardown + report
9. ALWAYS finish by stopping the daemon: `node lib/recon/recon-session.mjs --stop` — even if a step above failed. Never leave an orphaned chromium.
10. Close the debug run (stamps it finished with the final coverage stats): `node lib/debug/trace-cli.mjs --close --run=<runId>`.
11. Print BOTH the coverage report and the fail-reason histogram:
    - `node lib/recon/report.mjs` — the coverage report: honest denominator (explored / discovered / unreachable / remaining), per-route controls with danger/effect/purpose, and the causal control→endpoint map. The header now carries a `Declared sections: X/N reached` line so a bare DRAINED can never read "done" while declared sections are still pending.
    - `node lib/recon/report.mjs --route-coverage` — the ROUTE-LEVEL completeness oracle: `reached of N declared sections reached · Y unreachable (redirect/404) · Z pending`, and the LIST of the declared sections NOT yet reached. This is the honest answer to "did we see the whole app". A large `pending`/`unreachable` here (deep-linkable sections the router declares but a direct nav did not render) is the residual for the event-driven in-app-nav increment, NOT a claim of completeness.
    - `node lib/recon/report.mjs --unreached` — the fail-reason histogram: of everything NOT exercised, WHY (buckets: `unexplored` / `not-visible` / `NO_INSTANCE` / `REVEAL_*` / `danger-floor` / beyond-cap). This is the go/no-go artifact. A clean DRAIN should leave only honest limits (danger-skips, opaque regions, beyond-cap opener siblings). A STALL, or a large `unexplored` bucket carrying the `pending-INC.3` flag, means the dominant gap is the POST-nav route-collapse class (a distinct location collapsed under one routeKey — the next increment), NOT budget. The histogram cannot split route-collapse from budget yet (no location identity beyond routeKey until INC.3); it flags the class honestly rather than guessing.
12. Summarize for the user: how many templates were explored vs discovered, the crawl's terminal state (DRAINED vs STALLED, with `progress.reason` and any `cappedRemainder` beyond-cap remainder), the top fail-reason buckets from `--unreached`, which controls are `unreachable-coldstart` (behind in-app state — a known Phase-1 boundary), any controls skipped as destructive/auth/payment, and the causal edges found. Point them again at the tokenized admin url from step 4 (`http://127.0.0.1:7666/?t=<token>`) to inspect the run visually (the admin keeps serving `state/runs/` until they stop the process or the session ends). This graph is the input to Phase-2 test-case design.

## Boundaries (state honestly, do not paper over)
- Multi-route, same-origin: recon follows same-origin navigation and maps every page reachable by a DIRECT navigation, attributing each page's controls to its own route. A BFS ROUTE FRONTIER goes further than the agent's acts alone: `whats-new` harvests every same-origin `a[href]` at baseline + after each act into a queue (snapshot-only, edge-free), and `route-cli --visit-next` drains it — so a page beyond the element cap (a wide nav's 9th tab, a listing's detail page) is snapshotted even though no single control click reaches it. A url-pattern census bound visits ONE representative per pattern (a 1000-row listing → one detail route, the rest folded as siblings). Off-origin links are recorded (`external-link`) but never fired; the `/logout` danger route is never navigated.
- Declared-route denominator (route-manifest seeder, step 6c): on a constant-URL onClick SPA the a[href] harvest alone finds ~1 route, so the honest denominator is seeded from the app's OWN declared route list (extracted from its same-origin bundles) — DENOMINATOR + candidate seeding ONLY, never a coverage claim. Every declared section is still GENUINELY visited (or `markRouteUnreachable` on redirect/404). A deep-linkable section that renders on direct nav is now covered. A CONSTANT-URL onClick section (Groups/Events that swap content in place, URL unchanged) is now reached by the MENU-EVENT SWEEP: `dom-snapshot` stamps `node.navControl` on controls inside a `<nav>`/[role=navigation] landmark, `frontier` (navBatch) FRONT-LOADS them, and the agent acts each with `--opener-replayable` so the section hydrates + collects — its child reachable from a cold re-nav via reveal-replay (no `graph.states{}`). Residual, honestly `pending`/`unreachable`: a section reachable ONLY by a pushState that CHANGES the URL to a declared route whose direct goto renders an empty shell (reveal-replay refuses a route-changing hop), and a div-soup nav built from role-less `<div>`s OUTSIDE any `<nav>` landmark (reached reactively, not front-loaded). Also, `--route-coverage` credits a swept section as CONTROL coverage under `/dashboard`, not yet as its declared `/dashboard/groups` route (a report-time nav→route join is the follow-up). `:param` route patterns are counted but their concrete instances need within-section discovery (a later increment).
- Direct-navigation reach only: controls (or whole routes) revealed only after an in-app CLIENT-state action (a row's Edit after a search; an SPA view reachable only by clicking, not by URL) are DISCOVERED but recorded `unreachable-coldstart` — reaching them is the stay-on-page work, not yet built.
- Cookie/consent overlays are dismissed automatically before each snapshot/act (curated framework accept buttons + a consent-scoped accept-text fallback), so a consent wall no longer hides the controls beneath it. A fully custom banner the curated list misses still leaves its controls `not-visible` (honest, no silent skip).
- Login: recon can now authenticate via the optional "Authenticated mode" section above (a login pre-step persists a Playwright storageState the crawl loads). WITHOUT that section it maps only the surface reachable logged-out. Multi-step / SSO / OAuth / 2FA logins are NOT yet supported (single-form only); session-expiry mid-crawl is not auto-detected. Credentials come from the operator's environment, never `test.md` directly.
- Compose-collection (reveal-opener): a mutation-NAMED form-opener (`Create post` / `Compose` / `New message` / `Create group` / `Start chat`) is now OPENED to COLLECT its composer/modal as a READ (step 7b directive → `recon.md` step-4 `--reveal-opener`), so the create/messaging/group UIs a plain read-only walk skipped are mapped. The click's actual write is ABORTED by the session write-firewall (nothing committed); the SUBMIT control inside the form stays refused (never fired). A `Video Call`/`Go Live`/`Start Meeting` control is classed `communication` and HARD-refused (initiating a real-time call is an irreversible OUTWARD side-effect off the abortable HTTP layer) — mapped, never fired.
- Recon maps behavior; it does NOT fire destructive/logout/payment controls or fuzz inputs. That is later-phase work.
